# Wallet custody & recovery policy

SubScript operates **custodial** embedded wallets. This document is the recovery policy (checklist
item A2) and the resolution of the "mandatory wallet backup/export" item (A3).

## Custody models

Two backends sit behind one seam (`src/lib/custody/index.ts`). A wallet is resolved per-record:

| | Legacy (being retired) | Circle MPC (target) |
|---|---|---|
| Key location | Raw secp256k1 key, AES-encrypted at rest with `WALLET_ENCRYPTION_KEY` | Sharded across Circle's MPC; **no whole key exists anywhere** |
| Signing | Server decrypts the key into an `ethers.Wallet` | Circle developer-controlled API (`createContractExecutionTransaction`), authorized by the entity secret |
| Account type | EOA | SCA (ERC-4337); gas sponsored by Circle Gas Station on Arc |
| Selected when | `encrypted_private_key` present | `circle_wallet_id` present |

The migration off Legacy exists because `WALLET_ENCRYPTION_KEY` is a single secret that decrypts
*every* user's key — one leak is total loss. Circle MPC removes that single point of failure.

## Backup / export policy (A3 resolution)

**There is no user-facing private-key export for Circle wallets — by design.** MPC keys are not
extractable, so "back up your key" is not a meaningful action. `/api/user/wallet/export` refuses
Circle wallets (`canExportRawKey === false`). We do **not** add a blocking key-export step to
onboarding. "Backup" is reframed as **account recovery**:

- **User-facing recovery:** the account is bound to a verified email (OTP or Continue-with-Google).
  A user regains access to the same custodial wallet by re-authenticating that email — there is no
  seed phrase to lose.
- **Operator-facing recovery (the real root-key concern):** custody ultimately depends on the Circle
  **entity secret**. Its loss — not any user's — is the catastrophic case.

Legacy wallets remain exportable until they are swept to Circle (see below), after which export is
uniformly unavailable.

## Operator recovery — the entity secret

- The entity secret is generated by us and registered with Circle; registration produces a
  **recovery file**. Both live only in the secret manager (Vercel env + a vault), never in git.
- **Rotation:** generate a new entity secret, register its ciphertext (Circle keeps a short overlap),
  update `CIRCLE_ENTITY_SECRET`, and re-store the new recovery file. Old ciphertexts stop working
  after the overlap.
- **Loss:** losing *both* the entity secret and its recovery file means we can no longer authorize
  signing for existing Circle wallets. This is the single most important secret to protect and back
  up. Sandbox and production have **separate** entity secrets.

## Legacy → Circle migration (sweep)

Circle cannot import an external private key, so legacy wallets are migrated by value, not by key:

1. Provision a new Circle wallet for the user.
2. Using the legacy key one final time, sweep the wallet's USDC to the new Circle address.
3. Null the `encrypted_private_key` and set `circle_wallet_id`.
4. Once zero `encrypted_private_key` rows remain, delete the AES path and `WALLET_ENCRYPTION_KEY`.

Until a wallet is swept it keeps working on the legacy path (dual-run), so the migration is
incremental and reversible per-wallet.
